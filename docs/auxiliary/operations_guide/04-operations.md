# 04 - Operations runbook

## Logs

- App + Docker + Nginx + cloud-init: CloudWatch log group `/java-app/prod/app`.
- VPC flow logs: CloudWatch log group `/aws/vpc-flow-log/<vpc-id>`,
  created by the VPC module (prefix `/aws/vpc-flow-log/`, suffix is the
  VPC ID).
- ALB access logs: `s3://java-app-prod-alb-logs-<account>/AWSLogs/<account>/...`.
- RDS error/general/slowquery: exported to CloudWatch by RDS.

## Dashboards

- `java-app-prod-main` includes ALB request/5xx, ALB latency, ASG capacity,
  RDS CPU/connections/storage.

## Alarms

| Alarm name                          | Threshold              |
| ----------------------------------- | ---------------------- |
| `java-app-prod-alb-5xx`             | sum > 10 / 1m for 2m   |
| `java-app-prod-alb-unhealthy-targets` | > 0 for 2m            |
| `java-app-prod-rds-cpu`             | > 80% for 5m           |
| `java-app-prod-rds-free-storage`    | < 5 GiB for 10m        |
| `java-app-prod-rds-connections`     | > 150 for 5m           |
| `java-app-prod-ec2-disk`            | > 85% for 3m           |
| EventBridge -> SNS                  | ASG instance refresh failed/cancelled |

All actions go to SNS topic `java-app-prod-alarms`. Subscribe an email by
setting `var.alarm_email` and re-applying.

## Access an instance

```bash
INSTANCE=$(aws ec2 describe-instances \
  --filters "Name=tag:aws:autoscaling:groupName,Values=java-app-prod-asg" "Name=instance-state-name,Values=running" \
  --query 'Reservations[].Instances[].InstanceId' --output text | awk '{print $1}')
aws ssm start-session --target $INSTANCE
```

No SSH ingress is configured. Session Manager is the only access path.

## Force a release rollover

```bash
gh workflow run app-deploy.yml
```

Or manually:

```bash
aws ssm put-parameter --name /java-app/prod/backend-image-tag  --type String --overwrite --value sha-XXXX
aws ssm put-parameter --name /java-app/prod/frontend-image-tag --type String --overwrite --value sha-XXXX
aws ssm put-parameter --name /java-app/prod/release-id         --type String --overwrite --value "<rid>"

aws autoscaling start-instance-refresh \
  --auto-scaling-group-name java-app-prod-asg \
  --preferences '{"MinHealthyPercentage":100,"MaxHealthyPercentage":200,"InstanceWarmup":180,"AutoRollback":true}'
```

## Re-apply after a destroy (pending-deletion secrets)

When `infra-apply.yml` is run within 7 days of an `infra-destroy.yml`, it
fails creating the four app secrets:

```text
InvalidRequestException: You can't create this secret because a secret with
this name is already scheduled for deletion.
  with aws_secretsmanager_secret.db_app_user (and .admin, .jwt, .ses)
```

Cause: each secret in `infra/envs/prod/secrets.tf` is created with
`recovery_window_in_days = 7` (the AWS minimum). After `terraform destroy`
the secrets remain in `PendingDeletion` for that window, and AWS rejects
new secrets created with the same name until the window expires or the
pending secret is force-deleted.

Standard unblock - re-dispatch `infra-apply.yml` with the opt-in toggle:

```bash
gh workflow run infra-apply.yml -f purge_pending_secrets=true
```

What the toggle does: runs `.github/scripts/purge_pending_secrets.sh` after
AWS credentials are configured and before `terraform init`. The script
iterates the four well-known secret paths, calls `describe-secret`, and
only when `DeletedDate` is set issues
`delete-secret --force-delete-without-recovery`. Healthy or absent secrets
are skipped. The toggle defaults to `false`, so normal applies are
unaffected.

Equivalent local fallback (only when the workflow itself cannot run):

```bash
for s in \
  /java-app/prod/db/app-user \
  /java-app/prod/admin \
  /java-app/prod/jwt \
  /java-app/prod/ses
do
  aws secretsmanager delete-secret \
    --secret-id "$s" \
    --force-delete-without-recovery
done
```

Note: force-deletion is irrevocable. Acceptable here because each secret
is regenerated by Terraform on the next apply (`random_password` for
db-app-user/admin/jwt; deterministic config JSON for ses). Do not use this
procedure on long-lived production secrets that hold values not
reproducible from Terraform inputs.

## DB access

App connects with the least-privilege user from `/java-app/prod/db/app-user`.
Operators use the master credential from the RDS-managed master secret only
for one-off admin tasks; never bake the master credential into config.

## SES sandbox

Until production access is granted, SES will only deliver to verified
recipients. Verify a test recipient address in the SES console for early
validation.
